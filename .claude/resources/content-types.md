# Web Content Types

Reference guide for Red Hat web content types. These definitions help content creators use consistent terminology and select the appropriate content type for conveying information to users and customers.

**Source:** Red Hat Content Design and Planning - Content Types
**Last updated:** From internal The Source page

---

## Technical Content Types (Learn Category)

### Product Documentation*

**Definition:**
Modular, topic-based content that supports the use of a product, service, or technology. Product documentation is a release requirement and is carefully vetted by QE, SMEs, and other stakeholders.

Modular documentation is based on units of content called **modules**, which authors combine into organizational units called **assemblies**. A module is a self-contained unit of content that is based on one of three officially supported content types: **concept**, **task**, or **reference**. An assembly is a collection of several modules that the author creates to document a user story.

Product documentation covers end-to-end usage of an offering, from initial onboarding through to productive use, maintenance, and troubleshooting. Product documentation is more comprehensive and expected to be more stable than other types of content (for example, technical blog posts).

**Purpose:**
- Sets expectations of what the product, service, or technology can do
- Guides users through tasks: What do I need to do, why do I need to do it, and how do I do it?
- Enables informed decision making when evaluating the offering
- Helps users troubleshoot and recover from mistakes
- Provides a comprehensive, trustworthy reference source for information that users look up on demand
- Offers more in-depth content that supports release notes, in-UI/CLI content, nurture emails

**Content journey phases:** Discover, Learn, Evaluate, Adopt, Expand

**Owners:** CCS (Customer Content Services)

**Contributors:** Product management, engineering, support

**Style guide:** IBM Style guide, Red Hat Supplementary Style guide

**Format/Publishing:** Published for all Red Hat products as standalone guides (titles) in HTML format on the Red Hat Customer Portal. Most also available in PDF format.

**Related content types:** Release notes, Developer documentation

**Examples:**
- Developing a data model in Red Hat OpenShift Data Science
- Managing account access in Red Hat OpenShift Service Registry

### Developer Documentation*

**Definition:**
Technical reference information with usage examples specifically for the developer persona. It supplements and extends product documentation with other content types, for example, sample code that developers can run, examples, try-it-out API schemas.

**Purpose:**
- Describes the interfaces, libraries, and other reference information that are available to developers to use, build on, or interact with the product or service
- Provides examples and samples that help the developer to build applications, integrate, customize, and extend products and services
- Enables developers to successfully copy and paste from reference documentation so that they can quickly achieve their goal

**Content journey phases:** Learn, Evaluate, Adopt, Expand

**Owners:** Engineering

**Contributors:** Engineering, Customer Content Services

**Style guide:** IBM Style guide, PatternFly guidelines

**Format/Publishing:** Javadocs, API reference documentation, API schema or specification, README, code comments, database definition, sample code, articles/blogs, sample applications, code examples. API reference documentation, API schemas or specifications, Javadocs tend to be autogenerated using API documentation tools such as Swagger, Postman.

**Related content types:**
- Microcopy for API
- Microcopy for CLI
- Product documentation focused on Developers

**Examples:**
- Red Hat JBoss Enterprise Application Platform 7.4.0.GA public Javadoc
- API catalog and documentation (developers.redhat.com)

### Tutorial*

**Definition:**
A tutorial is a learning path designed to teach users how to use a product or perform a specific task. It provides step-by-step instructions, often with examples and hands-on exercises.

**Purpose:**
- Teaches users how to accomplish specific tasks
- Provides guided learning experience
- Includes practical examples and exercises
- Builds skills progressively

**Content journey phases:** Learn, Evaluate, Adopt

**Related content types:** Product documentation, Learning path, Quick start

---

### Quick Start*

**Definition:**
Quick starts provide users with step-by-step instructions to accomplish a specific goal in a short time period (for example, onboard a service, configure a specific feature, or install components).

There are two types of Quick starts:
- **GUI Quick starts:** Embedded in the graphical user interface (such as a side panel) to allow users to remain in context while following the instructions
- **CLI Quick starts:** Available as projects that contain source code and configuration files to help users achieve their goals and automate specific tasks

Quick starts often have the following characteristics:
- Mitigate the risk of mistakes by including a validation check in each task
- Highlight the prerequisites for both the entire Quick start and for each task
- Include an approximate time for completion
- Are available on demand, but the user can dismiss and resume at any time
- Save user progress; when relaunched, they open to last-read content

**Purpose:**
- Encourages confidence in the product or service, and autonomy in users
- Quickly engages users, and explains the value of a given application
- Reduces the time needed to start and complete key tasks, by keeping users in the UI and offering a concise set of context-specific steps and info
- Provides a jumping-off point for the next set of tasks or learning more
- Lowers demand for support by making answers to common questions readily available in the UI

**Content journey phases:** Discover, Learn, Evaluate, Adopt, Expand

**Owners:** CCS, UXD, Engineering (can vary by team)

**Contributors:** Subject matter experts (can vary by team)

**Style guide:** Best practices for writing quick starts (UXD Hub), IBM Style guide, PatternFly guidelines

**Format/Publishing:** Published in the UI using PatternFly

---

### Learning Path*

**Definition:**
A learning path is a curated series of content that guides users through a specific learning journey. It combines multiple content types (tutorials, documentation, videos, etc.) in a logical sequence.

**Purpose:**
- Provides structured learning experience
- Guides users from beginner to advanced knowledge
- Combines multiple content types for comprehensive learning
- Supports progressive skill building

**Content journey phases:** Learn, Evaluate, Adopt, Expand

**Related content types:** Tutorial, Product documentation, Quick start, Video

---

### FAQ*

**Definition:**
FAQs provide answers to frequently asked questions about a Red Hat product, solution, or service, or a deep dive into a single offering—for example, a training program.

**Purpose:**
Varies depending on content (business or technical). FAQs are evolve-loop content for Knowledgebase articles.

**Examples:**
- Dev sandbox FAQ (developers.redhat.com)
- ROSA security FAQ (redhat.com)

---

### In-App Content*

**Definition:**
In-app content is an umbrella concept made up of multiple content types such as Microcopy for UI, Tutorial, Release Notes and more.

Provides just enough guidance in the application to help users in the UI complete a specific task, reach a specific goal, or understand concepts.

(Optional) In-app help topics can include a list of links to relevant information. For example, you might want to link to the product documentation for details.

**Purpose:**
- Builds user's confidence about product/service and their ability to successfully complete tasks
- Keeps users in their current context with the just-in-time, just-enough guidance to get started and make informed decisions while completing the tasks
- Offers an easy path to find more detailed information or a logical next step
- Enables users that may typically avoid traditional documentation
- It is a self-service model of support

**Content journey phases:** Learn, Evaluate, Adopt, Expand

**Owners:** CCS, UXD

**Contributors:** CCS, UXD, Engineering

**Style guide:** IBM Style guide, PatternFly guidelines

## Supporting Content Types

### Release Notes

**Definition:**
Release notes document changes, new features, bug fixes, and known issues for a specific product release version.

**Purpose:**
- Informs users about what's new in the release
- Documents breaking changes and migration requirements
- Lists known issues and workarounds
- Provides version-specific information

**Content journey phases:** Adopt, Expand

**Related content types:** Product documentation

---

### Microcopy for UI

**Definition:**
Short, contextual text that appears in user interfaces to guide users through tasks and interactions.

**Purpose:**
- Provides in-context guidance
- Explains UI elements and actions
- Helps users complete tasks efficiently
- Reduces cognitive load

**Style guide:** IBM Style guide, PatternFly guidelines

**Examples:** Button labels, form field descriptions, tooltips, error messages, confirmation dialogs

---

### Microcopy for CLI and API

**Definition:**
Command-line interface help text and API documentation strings.

**Purpose:**
- Provides command usage information
- Documents parameters and options
- Explains API endpoints and methods
- Enables self-service discovery

**Style guide:** IBM Style guide

**Examples:** CLI help output, man pages, API endpoint descriptions, parameter documentation

---

### Microcopy for Announcements and Notifications

**Definition:**
Brief messages that inform users about system status, updates, or actions required.

**Purpose:**
- Communicates important information
- Alerts users to changes or issues
- Prompts user action when needed
- Maintains user awareness

**Examples:** System notifications, update announcements, maintenance alerts

## Choosing the Right Content Type

### For Technical Documentation:

- **Comprehensive product information:** Use **Product documentation** (modular: concept, task, reference modules combined into assemblies)
- **Developer-specific technical reference:** Use **Developer documentation** (API docs, code samples, Javadocs)
- **Version-specific changes:** Use **Release notes**
- **Learning and training:** Use **Tutorial** or **Learning path**
- **Quick task completion:** Use **Quick start**
- **Common questions:** Use **FAQ**
- **In-UI guidance:** Use **In-app content** or **Microcopy for UI**
- **CLI/API help:** Use **Microcopy for CLI and API**

### Content Type vs Module Type:

For **Product documentation** (the primary technical content type), use the modular documentation structure:
- **Concept modules** (con-*.adoc): Explain what and why
- **Procedure modules** (proc-*.adoc): Provide step-by-step how-to instructions
- **Reference modules** (ref-*.adoc): Present lookup data (commands, configurations, options)
- **Assemblies** (assembly-*.adoc): Combine modules to address user stories

### When to Use Different Types:

| User Need | Content Type | Module Type (if applicable) |
|-----------|--------------|----------------------------|
| Understand a concept | Product documentation | Concept module |
| Complete a task | Product documentation | Procedure module |
| Look up reference data | Product documentation | Reference module |
| Build with APIs | Developer documentation | N/A (API-specific) |
| Learn step-by-step | Tutorial or Quick start | N/A (standalone) |
| Find quick answers | FAQ | N/A (Q&A format) |
| Get in-UI help | In-app content | N/A (UI-specific) |
| See what changed | Release notes | N/A (version-specific) |

---

**Note:** Content types marked with "*" are in the "Learn" category and prioritized for helping users learn and adopt Red Hat products and services.
