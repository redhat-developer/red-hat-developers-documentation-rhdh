= Configure OpenTelemetry for SonataFlow services

[role="_abstract"]
Configure OpenTelemetry (OTel) to export trace data from your SonataFlow services to an OTLP-compatible backend. This allows you to monitor workflow execution paths and identify performance bottlenecks in your orchestration logic.

.Prerequisites

* A running OTLP-compatible collector (for example, Jaeger, Tempo, or OpenTelemetry Collector).

* Access to the service configuration (typically `application.properties` or environment variables).

.Procedure

. Open the configuration file for your SonataFlow service.

. Define the service name to identify the traces in your observability platform:
+
[source,yaml]
----
quarkus.otel.service.name=sonataflow-orchestrator
----
. Set the OTLP collector endpoint:
+
[source,yaml]
----
quarkus.otel.exporter.otlp.endpoint=http://<collector-hostname>:4317
----
. (Optional) Change the propagation format if using a specific service mesh (for example, B3 for older Zipkin installations):
+
[source,yaml]
----
quarkus.otel.propagators=tracecontext,baggage
----
. Enable the telemetry exporter:
+
[source,yaml]
----
quarkus.otel.enabled=true
----

.Verification

. Execute a workflow instance.

. Log in to your telemetry backend (for example, Jaeger).

. Search for the service `sonataflow-orchestrator` and verify that spans for the workflow execution appear in the timeline.
