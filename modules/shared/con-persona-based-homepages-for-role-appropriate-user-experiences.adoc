= Persona-based homepages for role-appropriate user experiences

[role="_abstract"]
Persona-based homepages provide role-specific default layouts based on RBAC group membership. Users see homepage cards aligned with their permissions, reducing context switching and improving daily workflow efficiency.

Platform engineers configure homepage layouts for specific user personas by attaching homepage configurations to RBAC groups.
When a user logs in, the backend API uses RBAC group membership to determine which homepage cards appear through visibility rule processing.
Users in multiple RBAC groups inherit homepage configurations from all groups they belong to, creating a combined homepage experience.

== How persona-based homepages differ from user customization

Persona-based homepages and user customization serve different purposes and can work together:

* *Persona-based homepages*: Platform engineers define default homepage layouts for specific user roles by using RBAC groups.
Users automatically see role-appropriate cards when they log in.
This ensures consistent experiences for users in the same role and prevents users from seeing cards they cannot use due to insufficient permissions.

* *User customization*: Individual users can customize their personal homepage layout by adding, removing, or rearranging cards.
Users can save custom layouts if the platform engineer has enabled user customization.
Personal customization overrides the persona-based defaults.

Platform engineers can configure both features simultaneously.
Users receive persona-appropriate defaults and can optionally customize their personal view.

== How persona-based homepages integrate with RBAC

Persona-based homepage configuration uses the same RBAC system that controls feature permissions:

* *RBAC groups control homepage visibility*: Platform engineers attach homepage cards to RBAC groups by using visibility rules.
The same groups that control feature access (for example, `team/developers`, `team/managers`) also control which homepage cards users see.

* *Consistency between permissions and visibility*: Users see homepage cards for features they have permission to access.
Platform engineers can prevent confusion by hiding cards for features users cannot use due to RBAC restrictions.

* *Inheritance from multiple groups*: Users who belong to multiple RBAC groups inherit homepage configurations from all groups.
For example, a developer who is also a team lead sees cards from both `team/developers` and `team/team-leads` groups.

== Common use cases

Platform engineers can create persona-specific homepages for different user roles:

* *Developer persona*: Homepage cards for software templates, starred catalog entities, deployment pipelines, and development documentation.
Developers can quickly create new components and access their frequently used services.

* *Data scientist persona*: Homepage cards for Jupyter notebooks, ML pipeline dashboards, data catalog searches, and model registry access.
Data scientists see tools relevant to machine learning workflows.

* *Manager persona*: Homepage cards for team health scorecards, approval workflows, portfolio health metrics, and engineering dashboards.
Managers see high-level overviews and team management tools.

* *Platform engineer persona*: Homepage cards for RBAC management, plugin registry access, system health monitoring, and infrastructure runbooks.
Platform engineers see administrative interfaces for managing {product-very-short}.

== Relationship to the homepage backend plugin

The homepage backend plugin forms the technical foundation for persona-based homepages:

* *Visibility evaluation*: The backend API processes visibility rules by using the logged-in user's context (RBAC groups, roles, permissions) to determine which homepage cards to include in the response.

* *Front-end displays cards*: The dynamic homepage front-end plugin requests default cards from the backend API and renders the cards that the response includes for the current user.

* *Manage configuration in `{my-app-config-file}`*: Platform engineers configure persona-based homepages by adding mount points with visibility rules to the `homepage.backend` configuration section.
