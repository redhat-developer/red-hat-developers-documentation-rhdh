1. Collect diagnostic data for support cases

When troubleshooting Red Hat Developer Hub issues or preparing a support case, you can collect comprehensive diagnostic data in one operation to reduce resolution time and provide complete information to Red Hat support teams.

1.1. Diagnostic data collection for support cases

When troubleshooting Red Hat Developer Hub deployment issues or preparing a support case, you can collect comprehensive diagnostic data in one operation to reduce investigation time and ensure complete information for support teams.

The RHDH must-gather tool is a specialized diagnostic data collection utility that targets Red Hat Developer Hub deployments on Red Hat OpenShift Container Platform (OpenShift Container Platform) and supported Kubernetes platforms including Microsoft Azure Kubernetes Service (AKS), Amazon Elastic Kubernetes Service (EKS), and Google Kubernetes Engine (GKE). Unlike generic must-gather tools, the RHDH must-gather tool includes collectors specific to RHDH components, such as Helm release data, Operator custom resources, Orchestrator workflows, and Backstage configurations.

The tool integrates with the OpenShift Container Platform must-gather framework on OpenShift Container Platform clusters, enabling familiar workflows for platform administrators. On Kubernetes platforms, it deploys as a Helm chart that runs a collection job and packages the results for download. Both methods offer the same comprehensive data collection capabilities, ensuring consistent diagnostic information regardless of platform.

You can use the RHDH must-gather tool in several scenarios:

  • When preparing to open a support case with Red Hat Global Support Services.
  • When troubleshooting production incidents that require detailed diagnostic information.
  • When conducting proactive health checks and configuration validation.
  • When debugging deployment issues in development or staging environments.

You can obtain the container image from quay.io/rhdh-community/rhdh-must-gather. Use an image version that matches your RHDH deployment version to ensure compatibility with your deployment method and platform.

1.2. Diagnostic data types and collection scope

To minimize security risk and configure appropriate collection scope for your environment, review what diagnostic data the tool gathers by default and what additional data you can include for specific troubleshooting scenarios.

The RHDH must-gather tool organizes data collection into default-enabled collectors and opt-in collectors.

1.2.1. Default-enabled collectors

The following collectors run by default and you can exclude them using --without-<collector> flags to focus collection on relevant data for your deployment:

Platform collector
This collector collects cluster-level information including cluster version, node details, storage classes, and platform detection metadata. This collector identifies whether the deployment is running on OpenShift Container Platform, AKS, EKS, or GKE, which helps support engineers understand platform-specific behaviors.
Helm collector
This collector gathers Helm release information for RHDH deployments installed using the Helm chart. This includes Helm release metadata, deployed values, and generated manifests. If your deployment uses the RHDH Operator instead of Helm, you can exclude this collector to focus on relevant data.
Operator collector
This collector collects Operator-specific data including operator logs, custom resource definitions (CRDs), and Backstage custom resources. This collector is essential for troubleshooting Operator-based deployments. If your deployment uses Helm installation instead of the Operator, you can exclude this collector.
Orchestrator collector
This collector gathers workflow execution data, Red Hat OpenShift Serverless resources, and Knative state for deployments by using the Orchestrator plugin. This collector is useful when troubleshooting workflow-related issues. If your deployment does not use Orchestrator functionality, you can exclude this collector to reduce collection time and output size.
Application data collector
This collector collects pod logs, deployment configurations, service definitions, ConfigMaps, and persistent volume claims for RHDH application components. This core diagnostic information helps in nearly all troubleshooting scenarios.
Route and Ingress collector
This collector gathers route definitions on OpenShift Container Platform and ingress configurations on Kubernetes platforms. This information is essential for troubleshooting external access issues and connectivity problems.
Namespace-inspect collector
This collector performs cross-namespace resource discovery to identify RHDH components distributed across many namespaces. This is particularly useful in complex deployments where RHDH resources span several namespaces. You can exclude this collector if you know all resources are in a single namespace.

1.2.2. Opt-in collectors

The following collectors do not run by default and you must explicitly enable them when needed:

Cluster-info collector
This collector collects cluster-wide state information beyond the basic platform metadata. Enable this collector when troubleshooting issues that might involve cluster-level configurations or resource constraints. This collector increases collection time and output size, so enable it only when cluster-level diagnostics are needed.
Secrets collector
This collector includes Kubernetes secrets in the diagnostic output with automatic redaction of sensitive values. The tool redacts passwords, tokens, API keys, and other sensitive data while preserving the structure and keys for configuration troubleshooting. The tool generates a redaction report showing which fields it redacted. Enable secrets collection only when troubleshooting configuration issues that require seeing secret structure, and always review the output before sharing with support.
Heap dumps collector
This collector triggers memory snapshots from Node.js backend processes and collects the resulting heap dump files. Heap dumps are large binary files that support engineers analyze to diagnose memory leaks, performance degradation, or out-of-memory errors. This collector requires that you configure backend containers with the NODE_OPTIONS environment variable including --heapsnapshot-signal=SIGUSR2. Enable this collector only when troubleshooting memory-related issues, as heap dump files can be several hundred megabytes each.

You can configure collection scope by excluding collectors not relevant to your deployment method, filtering by specific namespaces, and enabling optional collectors only when you need them. This customization reduces collection time, minimizes output size, and focuses diagnostic data on the areas relevant to your troubleshooting scenario.

1.3. Collect diagnostic data for OpenShift support cases

To collect Red Hat Developer Hub deployment data in one operation, use the OpenShift CLI (oc) oc adm must-gather command. This reduces troubleshooting time and ensures complete diagnostic information for support cases.

Prerequisites

  • OpenShift Container Platform 4.18 or later.
  • OpenShift CLI (oc) installed and authenticated to your cluster.
  • Cluster administrator access or appropriate RBAC permissions for resource inspection.
  • The RHDH deployment you want to collect diagnostics from is accessible.

Procedure

  1. Run the must-gather collection command: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest

    A collection pod starts, data collection begins, and results appear in a timestamped directory.

  2. Optional: Specify an image version that matches your RHDH deployment for compatibility: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:1.10.0

  3. Optional: Filter collection to specific namespaces if your RHDH deployment spans many namespaces: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --namespaces rhdh-operator,rhdh-app

    Replace rhdh-operator,rhdh-app with the comma-separated list of namespaces containing your RHDH resources.

  4. Optional: Exclude collectors not relevant to your deployment method to reduce collection time: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --without-helm

    Use --without-operator for Helm-based deployments, --without-helm for Operator-based deployments, or --without-orchestrator if not using Orchestrator functionality.

  5. Optional: Include cluster-wide information for troubleshooting cluster-level issues: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --cluster-info
  6. Monitor the collection progress. Status messages appear as data collection progresses through different collectors. Collection typically completes within 2-5 minutes, depending on the number of resources and optional collectors enabled.

Verification

  1. Locate the output directory. The output appears in ./must-gather.local.<timestamp> in your current working directory.

  2. Verify the expected directory structure: 

    $ ls must-gather.local.<timestamp>

    The output includes directories corresponding to the collectors that ran, such as helm/, operator/, and namespaces/.

  3. Verify the tool collected resource files and logs: 

    $ find must-gather.local.<timestamp> -name "*.log" | head -5
$ find must-gather.local.<timestamp> -name "*.yaml" | head -5

Troubleshooting

If the collection fails with permission errors
Verify you have cluster administrator access or the required RBAC permissions to read cluster resources, pod logs, and namespace information.
If the collection times out
The default timeout is typically enough for most deployments. If you have several namespaces or resources, filter to specific namespaces by using the --namespaces flag.
If specific collectors fail
Check the must-gather output for error messages. Some collectors depend on specific resources existing. For example, the Operator collector requires Backstage custom resources, which exist only in Operator-based deployments.

1.4. Collect diagnostic data for Kubernetes support cases

Deploy a Helm chart to collect diagnostic data from Microsoft Azure Kubernetes Service (AKS), Amazon Elastic Kubernetes Service (EKS), or Google Kubernetes Engine (GKE) deployments in one operation. This reduces troubleshooting time and ensures complete diagnostic information for support cases.

Prerequisites

  • Kubernetes cluster running on AKS, EKS, or GKE.
  • kubectl CLI authenticated to your cluster.
  • helm CLI 3 or later installed.
  • Cluster administrator access or appropriate RBAC permissions for resource inspection.
  • The RHDH deployment you want to collect diagnostics from is accessible.

Procedure

  1. Install the must-gather Helm chart: 

    $ helm install my-rhdh-must-gather redhat-developer-hub-must-gather \
  --repo https://redhat-developer.github.io/rhdh-chart \
  --namespace rhdh-diagnostics \
  --create-namespace

    A namespace for diagnostic collection is created and a Deployment resource starts to run the must-gather collection.

  2. Optional: Configure collection scope using Helm values. Create a values file to customize what data the tool collects: 

    mustGather:
  namespaces: "rhdh-app,rhdh-plugins"  # Comma-separated list
  excludeCollectors:
    - operator  # Exclude for Helm-based deployments
    - orchestrator  # Exclude if not using Orchestrator
  includeClusterInfo: true  # Include cluster-wide state

    Install with the values file: 

    $ helm install my-rhdh-must-gather redhat-developer-hub-must-gather \
  --repo https://redhat-developer.github.io/rhdh-chart \
  --namespace rhdh-diagnostics \
  --create-namespace \
  -f values.yaml

  3. Wait for the must-gather pod to be ready: 

    $ kubectl wait --for=condition=ready pod \
  -l app.kubernetes.io/instance=my-rhdh-must-gather,app.kubernetes.io/component=gather \
  --timeout=3600s -n rhdh-diagnostics

    Collection typically completes within 2-5 minutes, depending on the number of resources and optional collectors enabled.

  4. Extract the collected diagnostic data from the pod: 

    $ kubectl exec deploy/my-rhdh-must-gather -c data-holder -n rhdh-diagnostics -- tar czf - -C /must-gather . > rhdh-must-gather-output.tar.gz

    This command streams the diagnostic data archive directly to your local file system.

  5. Clean up the must-gather resources: 

    $ helm uninstall my-rhdh-must-gather -n rhdh-diagnostics
$ kubectl delete namespace rhdh-diagnostics

Verification

  1. Extract the archive: 

    $ tar -xzf rhdh-must-gather-output.tar.gz

  2. Verify the output directory structure: 

    $ ls must-gather/

    The output includes directories corresponding to the collectors that ran, such as helm/, namespaces/, and platform-specific directories.

  3. Verify the tool collected resource files and logs: 

    $ find must-gather -name "*.log" | head -5
$ find must-gather -name "*.yaml" | head -5

Troubleshooting

If the job fails to complete

Check the job status and pod logs for error messages: 

$ kubectl describe job -n rhdh-diagnostics
$ kubectl logs -n rhdh-diagnostics <pod_name>
If permission errors occur during collection
Verify the service account used by the must-gather job has appropriate RBAC permissions. Installing the chart creates necessary role bindings, but some cluster configurations might require additional permissions.
If the job times out
The default job timeout is typically enough for most deployments. If you have several namespaces or resources, configure the Helm values to filter collection to specific namespaces.
If Helm repository access fails
Ensure network connectivity to https://redhat-developer.github.io/rhdh-chart.

1.5. Filter data scope to reduce collection time

To reduce collection time and focus on relevant troubleshooting data, you can customize diagnostic collection scope by filtering namespaces, excluding deployment-method-specific collectors, and enabling optional data when needed.

Prerequisites

  • Understanding of your RHDH deployment architecture.
  • Knowledge of which namespaces contain RHDH resources.
  • Familiarity with your deployment method (Helm or Operator).
  • Basic understanding of must-gather collection on your platform (OpenShift Container Platform or Kubernetes).

Procedure

  1. Identify your deployment method to find which collectors are relevant. If you deployed RHDH using Helm, the Operator collector is not relevant. If you deployed using the Operator, the Helm collector is not relevant.
  2. Find whether you use Orchestrator functionality. If your deployment does not include the Orchestrator plugin or workflow capabilities, you can exclude the Orchestrator collector.

  3. Filter collection to specific namespaces in multi-namespace deployments:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --namespaces namespace1,namespace2,namespace3

    On Kubernetes (using Helm values file): 

    mustGather:
  namespaces: "namespace1,namespace2,namespace3"

    Replace namespace1,namespace2,namespace3 with the comma-separated list of namespaces containing your RHDH resources.

  4. Exclude collectors not relevant to your deployment:

    For Helm-based deployments, exclude the Operator collector:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --without-operator

    On Kubernetes (using Helm values file): 

    mustGather:
  excludeCollectors:
    - operator

    For Operator-based deployments, exclude the Helm collector:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --without-helm

    On Kubernetes (using Helm values file): 

    mustGather:
  excludeCollectors:
    - helm

    To exclude the Orchestrator collector:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --without-orchestrator

    On Kubernetes (using Helm values file): 

    mustGather:
  excludeCollectors:
    - orchestrator

    You can combine many exclusions:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --without-operator --without-orchestrator

    On Kubernetes (using Helm values file): 

    mustGather:
  excludeCollectors:
    - operator
    - orchestrator

  5. Optional: Include cluster-wide information when troubleshooting issues that might involve cluster-level configurations:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --cluster-info

    On Kubernetes (using Helm values file): 

    mustGather:
  includeClusterInfo: true

    The cluster-info collector gathers additional cluster state beyond basic platform metadata. This increases collection time and output size, so enable it only when you need cluster-level diagnostics.

  6. Optional: Configure timeouts for long-running operations in large deployments. The default timeouts are typically enough, but you can adjust them if needed:

    On OpenShift Container Platform (using environment variables in the must-gather pod spec): 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather

    On Kubernetes (using Helm values file): 

    mustGather:
  env:
    - name: COMMAND_TIMEOUT
      value: "60000"  # milliseconds

    The COMMAND_TIMEOUT environment variable controls how long collectors wait for commands to complete. The default varies by collector.

  7. Optional: Set log verbosity for detailed diagnostic output during collection:

    On Kubernetes (using Helm values file): 

    mustGather:
  env:
    - name: LOG_LEVEL
      value: "DEBUG"

    Valid values are DEBUG, INFO, WARN, and ERROR. The default is INFO. Use DEBUG when troubleshooting must-gather collection issues.

  8. Combine configuration options for focused collection. Example for a Helm-based deployment in specific namespaces with cluster info:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --namespaces rhdh-prod,rhdh-plugins --without-operator --cluster-info

    On Kubernetes (using Helm values file): 

    mustGather:
  namespaces: "rhdh-prod,rhdh-plugins"
  excludeCollectors:
    - operator
  includeClusterInfo: true

Verification

After collection completes, verify that the output has only the expected data:

  1. Check which collector directories exist in the output: 

    $ ls must-gather.local.<timestamp>/  # On OpenShift
$ ls must-gather-output/             # On Kubernetes
  2. Verify the output does not include excluded collectors. For example, if you excluded the Operator collector, the output should have no operator/ directory.

  3. If you filtered by namespaces, verify that only the specified namespaces appear in the namespaces/ directory: 

    $ ls must-gather.local.<timestamp>/namespaces/  # On OpenShift
$ ls must-gather-output/namespaces/             # On Kubernetes

1.6. Include redacted secrets in diagnostic collection

When troubleshooting configuration issues, you can include Kubernetes secrets in diagnostic data collection with automatic redaction to minimize security risk while preserving structure and keys needed for analysis.

Prerequisites

  • Understanding of basic must-gather collection on your platform.
  • Knowledge of which configuration issues require seeing secret structure.
  • Cluster administrator access (secrets collection requires elevated permissions).
  • Approval from your security team if required by your organization’s policies.

Procedure

  1. Assess whether secrets collection is necessary for your troubleshooting scenario. Secrets collection is useful when:
  2. Debugging configuration issues related to external integrations (database connections, authentication providers, API credentials)
  3. Troubleshooting permission errors that mightbe caused by incorrect secret references or malformed secret data

  4. Verifying that secret keys match the expected configuration schema

    Do not include secrets if you are troubleshooting issues unrelated to configuration, such as performance problems or resource constraints.

  5. Understand the security considerations:
  6. The tool redacts sensitive values (passwords, tokens, API keys, certificates) using pattern matching on field names (for example, password, token, apiKey) and value formats (for example, API key patterns)
  7. The tool preserves secret keys and structure to enable configuration troubleshooting
  8. The tool decodes Base64-encoded values in secret data fields before pattern matching to detect encoded sensitive data
  9. A redaction report documents which fields the tool redacted

  10. Manually review the output before sharing it with support to verify no sensitive data remains

    Important

    The tool might not detect Base64-encoded values in fields without sensitive names. Fields named generically (for example, value, content) that contain encoded secrets require manual verification.

  11. Enable secrets collection:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --with-secrets

    On Kubernetes (using Helm values file): 

    mustGather:
  includeSecrets: true

  12. Optional: Combine secrets collection with other configuration options:

    On OpenShift Container Platform (collect secrets from specific namespaces only): 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --with-secrets --namespaces rhdh-prod

    On Kubernetes (using Helm values file): 

    mustGather:
  namespaces: "rhdh-prod"
  includeSecrets: true
  excludeCollectors:
    - operator  # Exclude for Helm deployments
  13. Wait for collection to complete. Collection with secrets takes the same amount of time as basic collection, as the tool performs redaction during collection.

  14. Locate the redaction report in the output directory: 

    $ find must-gather.local.<timestamp> -name "redaction-report.txt"  # On OpenShift
$ find must-gather-output -name "redaction-report.txt"             # On Kubernetes

  15. Review the redaction report to understand what the tool redacted: 

    $ cat must-gather.local.<timestamp>/redaction-report.txt  # On OpenShift
$ cat must-gather-output/redaction-report.txt             # On Kubernetes

    The report lists the secret names and fields the tool redacted, along with the redaction patterns the tool applied.

  16. Manually review collected secrets before sharing with support: 

    $ find must-gather.local.<timestamp> -path "*/secrets/*" -name "*.yaml"  # On OpenShift
$ find must-gather-output -path "*/secrets/*" -name "*.yaml"             # On Kubernetes

    Open the secret files and verify that:

  17. Sensitive values show redaction markers like [REDACTED] or *
  18. No actual passwords, tokens, or API keys appear in the files

  19. The tool redacts or truncates certificate data

    If you find any residual sensitive values that automatic redaction did not catch, manually redact them before uploading.

  20. Optional: Search for environment-specific patterns that default redaction might not cover: 

    $ grep -r "mycompany.com" must-gather-local.<timestamp>/  # On OpenShift
$ grep -r "internal.network" must-gather-output/          # On Kubernetes

    Search for patterns that identify your organization or internal infrastructure that attackers could use for social engineering or targeted attacks. Internal service hostnames only visible within the cluster (for example, backstage.rhdh-prod.svc.cluster.local) typically do not need redaction. Redact domain names, external hostnames, or identifiers that attackers could use to identify your organization or facilitate attacks.

  21. Document what you included in your support case description:

    When attaching the must-gather output to your support case, note in the case description that you included secrets and that you have reviewed the redaction. This helps support engineers understand the available data and ensures proper handling.

Verification

  1. Verify that secret files are present in the output: 

    $ find must-gather.local.<timestamp> -path "*/secrets/*" -type f | wc -l  # On OpenShift
$ find must-gather-output -path "*/secrets/*" -type f | wc -l             # On Kubernetes

    The output includes at least one secret file for each namespace where RHDH stores configuration data.

  2. Verify the tool generated the redaction report: 

    $ cat must-gather.local.<timestamp>/redaction-report.txt | head -20  # On OpenShift
$ cat must-gather-output/redaction-report.txt | head -20             # On Kubernetes

  3. Spot-check a few secret files to confirm redaction: 

    $ cat must-gather.local.<timestamp>/namespaces/rhdh-prod/secrets/postgres-credentials.yaml

    The file includes keys like password, username, host with values marked as [REDACTED] or similar.

Troubleshooting

If the tool does not collect secrets
Verify you have cluster administrator permissions or appropriate RBAC to read secrets. The must-gather pod requires elevated privileges to access secrets across namespaces.
If the redaction report is missing
The tool generates the report only when you use the --with-secrets flag. Verify you enabled secrets collection in your must-gather command or Helm values.
If sensitive data appears in the output
The automatic redaction uses pattern matching and might not catch all custom or environment-specific sensitive data. Always manually review the output before sharing. If you find unredacted sensitive data, redact it manually or regenerate secrets after the support case is resolved.

1.7. Collect heap dumps to diagnose memory issues

When troubleshooting memory leaks or out-of-memory errors, you can collect Node.js heap dumps alongside diagnostic data to enable detailed memory analysis and accelerate root cause identification.

Prerequisites

  • Understanding of basic must-gather collection on your platform.
  • RHDH backend containers configured with the NODE_OPTIONS environment variable for heap dump support.
  • Sufficient storage for heap dump files (heap dumps can be several hundred megabytes each).
  • Cluster administrator access to modify deployment configurations.
  • Node.js version 12.0.0 or later (included in all Red Hat Developer Hub 1.10+ container images).

Procedure

  1. Verify whether you already configured your backend containers for heap dump support:

    For Operator-based deployments: 

    $ oc get backstage <backstage-cr-name> -n <namespace> -o jsonpath='{.spec.application.extraEnvs[?(@.name=="NODE_OPTIONS")].value}'

    For Helm-based deployments: 

    $ kubectl get deployment -n <namespace> <deployment-name> -o jsonpath='{.spec.template.spec.containers[*].env[?(@.name=="NODE_OPTIONS")].value}'

    If NODE_OPTIONS includes --heapsnapshot-signal=SIGUSR2, you already enabled heap dump support. If not, proceed to configure it.

  2. Configure the Node.js backend to accept heap dump signals:

    For Operator-based deployments, add the NODE_OPTIONS environment variable to your Backstage custom resource: 

    apiVersion: rhdh.redhat.com/v1alpha1
kind: Backstage
metadata:
  name: my-rhdh
spec:
  application:
    extraEnvs:
      - name: NODE_OPTIONS
        value: "--heapsnapshot-signal=SIGUSR2 --diagnostic-dir=/tmp"

    Apply the updated custom resource: 

    $ oc apply -f backstage-cr.yaml

    For Helm-based deployments, add the environment variable to your Helm values file: 

    upstream:
  backstage:
    extraEnvVars:
      - name: NODE_OPTIONS
        value: "--heapsnapshot-signal=SIGUSR2 --diagnostic-dir=/tmp"

    Update the Helm release: 

    $ helm upgrade my-rhdh redhat-developer-hub/backstage -f values.yaml -n <namespace>

  3. Wait for the backend pods to restart with the new configuration: 

    $ oc get pods -n <namespace> -w  # On OpenShift
$ kubectl get pods -n <namespace> -w  # On Kubernetes

  4. Verify the pods use the heap dump environment variable:

    For Operator-based deployments: 

    $ oc get pods -n <namespace> -l app.kubernetes.io/component=backstage -o jsonpath='{.items[0].spec.containers[*].env[?(@.name=="NODE_OPTIONS")].value}'

    For Helm-based deployments: 

    $ kubectl get pods -n <namespace> -l app.kubernetes.io/name=backstage -o jsonpath='{.items[0].spec.containers[*].env[?(@.name=="NODE_OPTIONS")].value}'

  5. Configure heap dump collection timeout if needed. Heap dump generation can take time, especially for backends with large memory footprints. The default timeout is 600 seconds (10 minutes):

    On OpenShift Container Platform (you cannot easily set environment variables with oc adm must-gather, so use default timeout): 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --with-heap-dumps

    On Kubernetes (using Helm values file): 

    mustGather:
  includeHeapDumps: true
  env:
    - name: HEAP_DUMP_TIMEOUT
      value: "900"  # seconds, increase if needed

  6. Enable heap dump collection:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --with-heap-dumps

    On Kubernetes (using Helm values file): 

    mustGather:
  includeHeapDumps: true

  7. Optional: Combine heap dump collection with other configuration options:

    On OpenShift Container Platform (collect heap dumps from specific namespaces): 

    $ oc adm must-gather --image=quay.io/rhdh-community/rhdh-must-gather:latest -- /usr/bin/gather --with-heap-dumps --namespaces rhdh-prod

    On Kubernetes (using Helm values file): 

    mustGather:
  namespaces: "rhdh-prod"
  includeHeapDumps: true
  excludeCollectors:
    - operator
  8. Monitor collection progress. Heap dump collection takes longer than basic collection, typically 5-15 minutes depending on the number of backend pods and their memory size. The must-gather tool sends a SIGUSR2 signal to each Node.js backend process, waits for the process to generate the heap snapshot, and collects the resulting files from the pod file system.

Verification

  1. Locate heap dump files in the output directory: 

    $ find must-gather.local.<timestamp> -name "*.heapsnapshot"  # On OpenShift
$ find must-gather-output -name "*.heapsnapshot"             # On Kubernetes

    The output includes at least one .heapsnapshot file for each RHDH backend pod running during collection.

  2. Check the size of the heap dump files: 

    $ find must-gather.local.<timestamp> -name "*.heapsnapshot" -exec ls -lh {} \;  # On OpenShift
$ find must-gather-output -name "*.heapsnapshot" -exec ls -lh {} \;             # On Kubernetes

    Heap dumps are typically 50-500 MB each, depending on the backend memory usage.

  3. Verify the total output size before uploading to a support case: 

    $ du -sh must-gather.local.<timestamp>  # On OpenShift
$ du -sh must-gather-output             # On Kubernetes

    Heap dumps can significantly increase the total output size. Ensure you have sufficient storage and network capacity to upload the archive to your support case.

Troubleshooting

If the system does not generate heap dumps

Verify you configured backend containers with the NODE_OPTIONS environment variable including --heapsnapshot-signal=SIGUSR2. Check the must-gather pod logs for errors related to signal delivery or snapshot generation: 

$ oc logs -n openshift-must-gather-<hash> <must-gather-pod>  # On OpenShift
$ kubectl logs -n rhdh-diagnostics <must-gather-pod>         # On Kubernetes
If collection times out during heap dump generation
Increase the HEAP_DUMP_TIMEOUT environment variable. Large backend memory footprints (multiple gigabytes) can take 10-15 minutes to snapshot. On OpenShift Container Platform, you may need to modify the must-gather image or use the Kubernetes Helm chart approach to set custom timeouts.
If heap dump files are empty or corrupted
Verify the Node.js version supports the --heapsnapshot-signal flag (Node.js 12+ required). Check backend pod logs for errors during snapshot generation.
If signal delivery fails
Verify that the must-gather service account has permissions to send signals to pods. The default must-gather RBAC should include this, but some cluster security policies may block it.

1.9.3. Log file naming conventions

Pod logs follow a consistent naming pattern to identify the pod and container that generated each log file: 

namespaces/<namespace-name>/pods/<pod-name>/logs/<container-name>.log

For example: 

namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/logs/backstage-backend.log

If a pod has multiple containers, each container has a separate log file: 

namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/logs/backstage-backend.log
namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/logs/oauth2-proxy.log

1.9.6. Heap dump file locations

When the tool collects heap dumps, they appear within pod directories in the namespace structure: 

namespaces/<namespace-name>/pods/<pod-name>/heapdumps/<timestamp>.heapsnapshot

For example: 

namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/heapdumps/20260430-143022-1.heapsnapshot

Heap dump files are binary files that can be analyzed using Chrome DevTools, Node.js heap analysis tools, or specialized memory profilers.

1.9.8. Using OpenShift Must-Gather Client for analysis

The OpenShift Must-Gather Client (OMC) is a command-line tool that simplifies analysis of must-gather output. OMC provides query capabilities and structured views of collected data.

To install OMC, run the following command: 

$ python3 -m pip install --user omc

To load must-gather data into OMC, run the following command: 

$ omc use must-gather.local.<timestamp>

Common OMC queries for RHDH diagnostics:

List all pods in the must-gather output: 

$ omc get pods

Show pod logs for a specific pod: 

$ omc logs <pod-name>

List all ConfigMaps: 

$ omc get configmaps

Filter resources by namespace: 

$ omc get pods -n rhdh-prod

Search logs for specific errors: 

$ omc logs -A | grep -i "error"

OMC provides a more efficient interface for navigating large must-gather outputs compared to manual file system navigation.

1.10.4. Disconnected and air-gapped environment support

The RHDH must-gather tool supports disconnected environments with image mirroring.

Disconnected environment requirements: * Mirror the must-gather image to the internal container registry. * Configure must-gather to use the mirrored image location. * Ensure network connectivity from the cluster to the internal registry.

To mirror the must-gather image, run the following commands: 

$ podman pull quay.io/rhdh-community/rhdh-must-gather:1.10.0
$ podman tag quay.io/rhdh-community/rhdh-must-gather:1.10.0 internal-registry.example.com/rhdh/must-gather:1.10.0
$ podman push internal-registry.example.com/rhdh/must-gather:1.10.0

To use the mirrored image on OpenShift, run: 

$ oc adm must-gather --image=internal-registry.example.com/rhdh/must-gather:1.10.0

To use the mirrored image on Kubernetes, configure the following Helm values: 

image:
  repository: internal-registry.example.com/rhdh/must-gather
  tag: "1.10.0"

Data transfer considerations for disconnected environments:

  • You must transfer collected diagnostic data from the disconnected environment to a system with internet access for support case upload.
  • Output directory size varies based on collection scope.
  • Heap dumps significantly increase output size (500 MB - 2 GB total).
  • Test collection scope in a non-production environment to estimate output size before production use.

1.10.5. Required RBAC permissions

The RHDH must-gather tool requires elevated permissions to read cluster resources, pod logs, and namespace information.

Minimum required permissions: * Read access to pods, deployments, services, configmaps in target namespaces. * Read access to pod logs. * Read access to custom resources (Backstage CRs for Operator deployments, Helm releases for Helm deployments). * Read access to routes (OpenShift Container Platform) or ingress resources (Kubernetes). * Read access to secrets (only when using --with-secrets). * Ability to execute commands in pods (only when using --with-heap-dumps to send signals).

On OpenShift Container Platform, the oc adm must-gather command typically runs with cluster-admin privileges, which include all necessary permissions.

On Kubernetes platforms, installing the must-gather Helm chart creates a service account with appropriate role bindings. If your cluster has restrictive security policies, you might need to create additional role bindings or use a service account with elevated permissions.

1.10.6. Known limitations and unsupported scenarios

The following scenarios have known limitations or lack support:

Unsupported platforms
Red Hat does not test or officially support Kubernetes distributions other than AKS, EKS, GKE, and OpenShift Container Platform. The tool might work on other Kubernetes distributions, but Red Hat does not guarantee functionality.
Multi-cluster deployments
The must-gather tool collects data from a single cluster. For RHDH deployments spanning multiple clusters, run must-gather separately on each cluster.
Dynamic plugin diagnostics
While the tool collects pod logs and configurations that include dynamic plugin state, it does not have specialized collectors for individual dynamic plugins. Plugin-specific diagnostics rely on application logs and ConfigMap data.
Ephemeral environments
The must-gather tool requires resources to remain stable during collection (2-15 minutes depending on scope). In highly dynamic environments with frequent pod restarts, the tool might collect incomplete data.

1.10.7. Platform-specific implementation notes

OpenShift Container Platform
The must-gather tool integrates with the OpenShift must-gather framework. The oc adm must-gather command creates a dedicated pod that runs the collection. OpenShift automatically cleans up the pod after collection completes. The tool writes output to the local file system in the must-gather.local.<timestamp> directory.
AKS
The must-gather tool deploys as a Kubernetes Job via Helm chart. Collection runs to completion, and you must extract the output from the pod by using kubectl cp. The job and pod resources remain after completion to allow data extraction, then you manually clean them up using helm uninstall.
EKS
Implementation is identical to AKS. Installing the Helm chart creates a Job that runs must-gather collection. Authentication to the EKS cluster uses kubectl with AWS credentials.
GKE
Implementation is identical to AKS. Installing the Helm chart creates a Job that runs must-gather collection. Authentication to the GKE cluster uses kubectl with GCP credentials.
