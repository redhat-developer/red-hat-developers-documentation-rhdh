Generate diagnostic data from Red Hat Developer Hub deployments on OpenShift Container Platform and Kubernetes platforms by using the must-gather tool to accelerate troubleshooting and provide complete information for support cases.

1. Generate diagnostic data using the must-gather tool to streamline support cases

When opening a support ticket for RHDH deployment issues, use the must-gather tool to quickly generate diagnostic data. The tool collects deployment manifests, logs, and resource definitions in one operation to resolve platform issues faster.

Run must-gather on OpenShift Container Platform using oc adm must-gather. On Kubernetes platforms (AKS, EKS, GKE), deploy a Helm chart.

1.1. When to use must-gather

Use must-gather when:

  • Opening a support ticket with Red Hat Global Support Services
  • Troubleshooting deployment, configuration, or runtime issues
  • Conducting health checks before upgrades
Note

Typical collection size:

  • Basic collection: 10-50 MB
  • With heap dumps: 500 MB - 2 GB

Plan for adequate storage and bandwidth when collecting for support tickets.

1.2. Quick start

On OpenShift Container Platform: 

$ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest

Output: ./must-gather.local.<timestamp>

On Kubernetes platforms (AKS, EKS, GKE):

  1. Install must-gather: 

    $ helm install my-rhdh-must-gather redhat-developer-hub-must-gather \
  --repo https://charts.openshift.io \
  --namespace rhdh-diagnostics \
  --create-namespace

  2. Wait for collection to complete: 

    $ kubectl wait --for=condition=ready pod \
  -l app.kubernetes.io/instance=my-rhdh-must-gather,app.kubernetes.io/component=gather \
  --timeout=3600s -n rhdh-diagnostics

  3. Download collected data: 

    $ kubectl exec deploy/my-rhdh-must-gather -c data-holder -n rhdh-diagnostics -- \
  tar czf - -C /must-gather . > rhdh-must-gather-output.tar.gz

  4. Clean up: 

    $ helm uninstall my-rhdh-must-gather -n rhdh-diagnostics

For configuration options, see the sections below.

1.3. Deployment methods and data collection

RHDH supports two deployment methods, and the must-gather tool uses different collectors for each method.

1.3.1. Operator-based deployments

Operator-based deployments use the RHDH Operator to manage Backstage custom resources. These deployments have Backstage custom resources in the cluster that the Operator controller watches and reconciles.

The must-gather Operator collector gathers:

  • Operator controller logs
  • Backstage custom resource definitions
  • Operator configuration and status

For Operator deployments, exclude the Helm collector to focus collection on relevant data.

1.3.2. Helm-based deployments

Helm-based deployments use the RHDH Helm chart to deploy Kubernetes resources directly without an Operator. These deployments have Helm releases that manage the application lifecycle.

The must-gather Helm collector gathers:

  • Helm release information and manifests
  • Deployed resource configurations
  • Helm chart values

For Helm deployments, exclude the Operator collector to focus collection on relevant data.

1.3.3. Mixed deployments

Clusters can contain both Operator-managed and Helm-managed RHDH instances in different namespaces. In this case, run must-gather without exclusion flags to collect data from all deployment methods.

1.4. Diagnostic data types and collection scope

The must-gather tool uses collectors to gather specific types of diagnostic data.

Reduce collection time by:

  • Excluding collectors not needed for your deployment
  • Filtering to specific namespaces
  • Enabling optional collectors only when required
Important

The must-gather output might contain sensitive information:

  • Configuration values and environment variables
  • Application logs and resource definitions

Security measures:

  • The tool automatically sanitizes known secret types
  • Always review output before sharing with support
  • Manually check for domain-specific sensitive data

1.4.1. Default-enabled collectors

The following collectors run by default. Exclude collectors not needed for your deployment using --without-<collector> flags:

platform collector
Collects cluster version and platform type (OpenShift Container Platform, AKS, EKS, or GKE). Disable with --without-platform.
helm collector
Gathers Helm release information. Use for Helm-based deployments only. Disable with --without-helm.
operator collector
Collects Operator logs and custom resources. Use for Operator-based deployments only. Disable with --without-operator.
orchestrator collector
Gathers workflow data for the Orchestrator plugin. Use only when troubleshooting Orchestrator workflows. Disable with --without-orchestrator.
route-ingress collector
Collects route definitions (OpenShift Container Platform) and ingress configurations (Kubernetes). Use when troubleshooting external access. Disable with --without-route or --without-ingress.
namespace-inspect collector
Organizes namespace resources for support teams. Limit to specific namespaces using --namespaces. Recommended for all collections. Disable with --without-namespace-inspect.

1.4.2. Opt-in collectors

The following collectors do not run by default. Enable them only when needed:

cluster-info collector

Collects cluster-wide state information beyond basic platform metadata. Enable with --with-cluster-info.

Important

Enable only when support requests it.

Impact:

  • Significantly increases collection time
  • May impact cluster performance during collection
heap-dumps collector

Collects Node.js memory snapshots for diagnosing memory issues. Enable with --with-heap-dumps.

Requirements:

  • Liveness probe timeout increased to prevent pod restarts
  • Sufficient storage (50-500 MB per heap dump file)

Use only when:

  • Troubleshooting memory leaks
  • Support requests memory analysis
  • Diagnosing out-of-memory errors

1.5. Run must-gather for Operator-based deployments to collect diagnostic data

Collect diagnostic data from Operator-based RHDH deployments.

Prerequisites

  • Cluster administrator access or RBAC permissions for resource inspection.
  • Authenticated CLI: OpenShift CLI (oc) (OpenShift Container Platform) or kubectl and helm (Kubernetes).

Procedure

  1. Verify you have an Operator-based deployment: 

    $ oc get backstage -A

    If this returns Backstage custom resources, you have an Operator deployment.

  2. Run must-gather:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --without-helm

    Output: ./must-gather.local.<timestamp>

    On Kubernetes (AKS, EKS, GKE):

    Install must-gather: 

    $ helm upgrade --install my-rhdh-must-gather redhat-developer-hub-must-gather \
  --repo https://charts.openshift.io \
  --namespace rhdh-diagnostics \
  --create-namespace \
  --set gather.withHelm=false

    Wait for collection: 

    $ kubectl wait --for=condition=ready pod \
  -l app.kubernetes.io/instance=my-rhdh-must-gather,app.kubernetes.io/component=gather \
  --timeout=3600s -n rhdh-diagnostics

    Extract data: 

    $ kubectl exec deploy/my-rhdh-must-gather -c data-holder -n rhdh-diagnostics -- \
  tar czf - -C /must-gather . > rhdh-must-gather-output.tar.gz

    Clean up: 

    $ helm uninstall my-rhdh-must-gather -n rhdh-diagnostics

Verification

Check the output directory exists:

On OpenShift Container Platform: 

$ ls must-gather.local.<timestamp>/operator/

On Kubernetes: 

$ tar -tzf rhdh-must-gather-output.tar.gz | grep operator/

1.7. Collect heap dumps with the inspector protocol to diagnose memory issues

Collect heap dumps using the inspector protocol (default method).

Prerequisites

  • Sufficient storage for heap dump files (50-500 MB each).
  • Your deployment does not use --disable-sigusr1 in NODE_OPTIONS.

Procedure

  1. Increase the liveness probe timeout to prevent pod restarts:

    Important

    RHDH stops responding during heap dump collection. Large memory footprints (1GB+) can cause liveness probe failures. Plan collection during maintenance windows or low-traffic periods.

    Operator deployments: 

    $ oc patch deployment <deployment-name> -n <namespace> \
  -p '{"spec":{"template":{"spec":{"containers":[{"name":"backstage-backend","livenessProbe":{"failureThreshold":180}}]}}}}'

    Helm deployments: 

    upstream:
  backstage:
    livenessProbe:
      failureThreshold: 180

    Wait for pods to restart before proceeding.

  2. Run must-gather with heap dumps enabled:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --with-heap-dumps

    On Kubernetes: 

    gather:
  heapDump:
    enabled: true
    method: "inspector"

    Collection takes 5-15 minutes depending on pod count and memory size.

Verification

Check for .heapsnapshot files: 

$ find must-gather.local.<timestamp> -name "*.heapsnapshot"  # On OpenShift
$ tar -tzf rhdh-must-gather-output.tar.gz | grep heapsnapshot  # On Kubernetes
If collection starts but times out before completing

You see progress messages but collection eventually times out before finishing. Check the must-gather logs for warnings about liveness probe settings: 

[WARN] Pod 'xxx' may restart during heap dump collection!
[WARN]   Current: failureThreshold=3 × periodSeconds=10s = 30s before restart
[WARN]   Required: at least 900s (HEAP_DUMP_TIMEOUT)

Set your liveness probe timeout longer than the collection time. Increase the timeout to allow collection to complete: 

$ oc patch deployment <deployment-name> -n <namespace> \
  -p '{"spec":{"template":{"spec":{"containers":[{"name":"backstage-backend","livenessProbe":{"failureThreshold":180}}]}}}}'

Wait for your pods to restart with the new configuration, then run heap dump collection again.

If collection times out

Large memory footprints (multiple gigabytes) can take 10-15 minutes to snapshot. Increase the heap dump timeout to allow more time for collection.

On OpenShift Container Platform, set the timeout using an environment variable: 

$ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest \
  -- /usr/bin/env HEAP_DUMP_TIMEOUT=1800 /usr/bin/gather --with-heap-dumps

On Kubernetes, set the timeout in your Helm values: 

gather:
  heapDump:
    enabled: true
    timeout: "1800"

1.8. Collect heap dumps with custom SIGUSR2 configuration to diagnose memory issues

Collect heap dumps using the SIGUSR2 signal method when the inspector protocol is not suitable.

Prerequisites

  • Sufficient storage for heap dump files (50-500 MB each).
  • Node.js 12+ required for --heapsnapshot-signal flag support.

Procedure

  1. Configure NODE_OPTIONS for SIGUSR2:

    Operator deployments: 

    apiVersion: rhdh.redhat.com/v1alpha1
kind: Backstage
metadata:
  name: my-rhdh
spec:
  application:
    extraEnvs:
      - name: NODE_OPTIONS
        value: "--heapsnapshot-signal=SIGUSR2 --diagnostic-dir=/tmp"

    Apply the change: 

    $ oc apply -f backstage-cr.yaml

    Helm deployments: 

    upstream:
  backstage:
    extraEnvVars:
      - name: NODE_OPTIONS
        value: "--heapsnapshot-signal=SIGUSR2 --diagnostic-dir=/tmp"

    Update the release: 

    $ helm upgrade my-rhdh redhat-developer-hub/backstage -f values.yaml -n <namespace>

    Wait for pods to restart.

  2. Increase the liveness probe timeout to prevent pod restarts:

    Important

    RHDH stops responding during heap dump collection. Large memory footprints (1GB+) can cause liveness probe failures. Plan collection during maintenance windows or low-traffic periods.

    Operator deployments: 

    $ oc patch deployment <deployment-name> -n <namespace> \
  -p '{"spec":{"template":{"spec":{"containers":[{"name":"backstage-backend","livenessProbe":{"failureThreshold":180}}]}}}}'

    Helm deployments: 

    upstream:
  backstage:
    livenessProbe:
      failureThreshold: 180

    Wait for pods to restart before proceeding.

  3. Run must-gather with SIGUSR2 method:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --with-heap-dumps --heap-dump-method sigusr2

    On Kubernetes: 

    gather:
  heapDump:
    enabled: true
    method: "sigusr2"

    Collection takes 5-15 minutes depending on pod count and memory size.

Verification

Check for .heapsnapshot files: 

$ find must-gather.local.<timestamp> -name "*.heapsnapshot"  # On OpenShift
$ tar -tzf rhdh-must-gather-output.tar.gz | grep heapsnapshot  # On Kubernetes

Troubleshooting

If heap dumps are not collected

Verify NODE_OPTIONS includes --heapsnapshot-signal=SIGUSR2 --diagnostic-dir=/tmp. Check pod logs: 

$ oc logs -n openshift-must-gather-<hash> <must-gather-pod>  # On OpenShift
$ kubectl logs -n rhdh-diagnostics <must-gather-pod>         # On Kubernetes
If collection starts but times out before completing

You see progress messages but collection eventually times out before finishing. Check the must-gather logs for warnings about liveness probe settings: 

[WARN] Pod 'xxx' may restart during heap dump collection!
[WARN]   Current: failureThreshold=3 × periodSeconds=10s = 30s before restart
[WARN]   Required: at least 900s (HEAP_DUMP_TIMEOUT)

Set your liveness probe timeout longer than the collection time. Increase the timeout to allow collection to complete: 

$ oc patch deployment <deployment-name> -n <namespace> \
  -p '{"spec":{"template":{"spec":{"containers":[{"name":"backstage-backend","livenessProbe":{"failureThreshold":180}}]}}}}'

Wait for your pods to restart with the new configuration, then run heap dump collection again.

If collection times out

Large memory footprints (multiple gigabytes) can take 10-15 minutes to snapshot. Increase the heap dump timeout to allow more time for collection.

On OpenShift Container Platform, set the timeout using an environment variable: 

$ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest \
  -- /usr/bin/env HEAP_DUMP_TIMEOUT=1800 /usr/bin/gather --with-heap-dumps --heap-dump-method sigusr2

On Kubernetes, set the timeout in your Helm values: 

gather:
  heapDump:
    enabled: true
    method: "sigusr2"
    timeout: "1800"

1.10. Diagnostic data output structure and organization

The must-gather output directory organizes data by collector type, namespace, and resource type.

1.10.1. Top-level directory structure

The must-gather output directory contains files and subdirectories organized by collection method and data type. The following table shows the top-level structure:

PathContents

version

Tool version information (for example, rhdh-must-gather x.y.z-sha).

sanitization-report.txt

Data sanitization summary and details.

all-routes.txt

All OpenShift Container Platform routes cluster-wide.

all-ingresses.txt

All Kubernetes ingresses cluster-wide.

must-gather.log

Must-gather container logs (if running in pod).

cluster-info/

Cluster-wide information (only present if you specified --with-cluster-info).

namespace-inspect/

Deep namespace inspect data (collected by default).

platform/

Platform and infrastructure information.

helm/

Helm deployment data (native releases and standalone deployments). Only present if Helm-deployed RHDH instances are detected.

orchestrator/

Orchestrator-flavored deployment data (if detected). Only present if Orchestrator components are detected.

operator/

Operator deployment data (if RHDH operators found). Only present if RHDH operators are detected.

1.10.2. Namespace directory structure

Within the namespace-inspect/namespaces/ directory, each namespace has a subdirectory containing resource definitions and logs organized by Kubernetes API group. The following table shows the per-namespace directory organization:

SubdirectoryContents

namespaces/<namespace-name>/pods/

Pod definitions and logs. Each pod has a subdirectory containing its YAML definition and log files for each container.

namespaces/<namespace-name>/apps/

Application workload resources including Deployments, StatefulSets, DaemonSets, and ReplicaSets.

namespaces/<namespace-name>/core/

Core Kubernetes resources including Services, ConfigMaps, Secrets, and PersistentVolumeClaims.

namespaces/<namespace-name>/networking.k8s.io/

Network resources including Ingresses and NetworkPolicies.

namespaces/<namespace-name>/route.openshift.io/

Route resources (OpenShift Container Platform only).

namespaces/<namespace-name>/batch/

Batch workload resources including Jobs and CronJobs.

namespaces/<namespace-name>/autoscaling/

Autoscaling resources including HorizontalPodAutoscalers.

Note

The directory structure follows the Kubernetes API group organization as generated by oc adm inspect namespace. Custom resources appear in subdirectories named after their API groups (for example, rhdh.redhat.com/ for Backstage resources).

1.10.3. Log file naming conventions

Pod logs follow a consistent naming pattern to identify the pod and container that generated each log file: 

namespace-inspect/namespaces/<namespace-name>/pods/<pod-name>/<container-name>/logs/current.log

For example: 

namespace-inspect/namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/backstage-backend/logs/current.log

If a pod has multiple containers, each container has a separate subdirectory with its own logs: 

namespace-inspect/namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/backstage-backend/logs/current.log
namespace-inspect/namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/oauth2-proxy/logs/current.log

Previous container logs are available as previous.log in the same directory.

1.10.6. Heap dump file locations

When the tool collects heap dumps with the --with-heap-dumps flag, they appear within deployment-specific directories, not under namespace-inspect/namespaces/. The location depends on the installation method:

For Helm releases: 

helm/releases/ns=<namespace>/<release-name>/deployment/heap-dumps/pod=<pod-name>/container=<container-name>/heapdump-<timestamp>.heapsnapshot

For Helm standalone deployments: 

helm/standalone/ns=<namespace>/<workload-name>/deployment/heap-dumps/pod=<pod-name>/container=<container-name>/heapdump-<timestamp>.heapsnapshot

For Operator-managed instances: 

operator/backstage-crs/ns=<cr-namespace>/<cr-name>/deployment/heap-dumps/pod=<pod-name>/container=<container-name>/heapdump-<timestamp>.heapsnapshot

For example, a Helm release heap dump: 

helm/releases/ns=rhdh-prod/developer-hub/deployment/heap-dumps/pod=backstage-developer-hub-7d8f9c5b-xk2m4/container=backstage-backend/heapdump-20260430-143022.heapsnapshot

Heap dump files are binary files that can be analyzed using Chrome DevTools, Node.js heap analysis tools, or specialized memory profilers.

1.11.4. Disconnected and air-gapped environment support

The RHDH must-gather tool supports disconnected environments with image mirroring.

Disconnected environment requirements: * Mirror the must-gather image to the internal container registry. * Configure must-gather to use the mirrored image location. * Ensure network connectivity from the cluster to the internal registry.

To mirror the must-gather image, run the following commands: 

$ podman pull registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:1.10.0
$ podman tag registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:1.10.0 internal-registry.example.com/rhdh/must-gather:1.10.0
$ podman push internal-registry.example.com/rhdh/must-gather:1.10.0

To use the mirrored image on OpenShift Container Platform, run: 

$ oc adm must-gather --image=internal-registry.example.com/rhdh/must-gather:1.10.0
Note

If your internal registry requires authentication, add the registry credentials to the cluster-wide pull secret in the openshift-config namespace. The oc adm must-gather command creates temporary namespaces, so you must use cluster-wide credentials rather than namespace-specific secrets.

To use the mirrored image on Kubernetes, configure the following Helm values: 

image:
  registry: internal-registry.example.com
  repository: rhdh/must-gather
  tag: "1.10.0"
Note

If your internal registry requires authentication, create a docker registry secret and reference it in the Helm values: 

$ kubectl create secret docker-registry my-registry-secret \
  --docker-server=internal-registry.example.com \
  --docker-username=<username> \
  --docker-password=<password> \
  --namespace rhdh-diagnostics

Then add the secret to your Helm values file: 

imagePullSecrets:
  - name: my-registry-secret

Data transfer considerations for disconnected environments:

  • You must transfer collected diagnostic data from the disconnected environment to a system with internet access for support case upload.
  • Output directory size varies based on collection scope.
  • Heap dumps significantly increase output size (500 MB - 2 GB total).
  • Test collection scope in a non-production environment to estimate output size before production use.

1.11.5. Required RBAC permissions

The RHDH must-gather tool requires elevated permissions to read cluster resources, pod logs, and namespace information.

Minimum required permissions: * Read access to pods, deployments, services, configmaps in target namespaces. * Read access to pod logs. * Read access to custom resources (Backstage CRs for Operator deployments, Helm releases for Helm deployments). * Read access to routes (OpenShift Container Platform) or ingress resources (Kubernetes). * Ability to execute commands in pods (only when using --with-heap-dumps to send signals).

On OpenShift Container Platform, oc adm must-gather typically requires cluster-admin privileges, which include all necessary permissions.

On Kubernetes platforms, installing the must-gather Helm chart creates a service account with appropriate role bindings. If your cluster has restrictive security policies, you might need to create additional role bindings or use a service account with elevated permissions.

1.11.6. Known limitations and unsupported scenarios

The following scenarios have known limitations or lack support:

Unsupported platforms
Red Hat does not test or officially support Kubernetes distributions other than AKS, EKS, GKE, and OpenShift Container Platform. The tool might work on other Kubernetes distributions, but Red Hat does not guarantee functionality.
Multi-cluster deployments
The must-gather tool collects data from a single cluster. For RHDH deployments spanning multiple clusters, run must-gather separately on each cluster.
Dynamic plugin diagnostics
While the tool collects pod logs and configurations that include dynamic plugin state, it does not have specialized collectors for individual dynamic plugins. Plugin-specific diagnostics rely on application logs and ConfigMap data.
Ephemeral environments
The must-gather tool requires resources to remain stable during collection (2-15 minutes depending on scope). In highly dynamic environments with frequent pod restarts, the tool might collect incomplete data.

1.11.7. Platform-specific implementation notes

OpenShift Container Platform
The must-gather tool integrates with the OpenShift must-gather framework. The oc adm must-gather command creates a dedicated pod that runs the collection. OpenShift automatically cleans up the pod after collection completes. The tool writes output to the local file system in the must-gather.local.<timestamp> directory.
AKS
The must-gather tool deploys as a Kubernetes Job via Helm chart. Collection runs to completion, and you must extract the output from the pod by using kubectl cp. The job and pod resources persist after completion to allow data extraction, then you manually clean them up using helm uninstall.
EKS
Implementation is identical to AKS. Installing the Helm chart creates a Job that runs must-gather collection. Authentication to the EKS cluster uses kubectl with AWS credentials.
GKE
Implementation is identical to AKS. Installing the Helm chart creates a Job that runs must-gather collection. Authentication to the GKE cluster uses kubectl with GCP credentials.
