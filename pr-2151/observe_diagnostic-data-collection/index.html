1. Generate diagnostic data using the must-gather tool to streamline support cases

Generate diagnostic data by using the must-gather tool to streamline support cases by collecting comprehensive diagnostic data in one operation to reduce resolution time and provide complete information to Red Hat support teams.

1.1. Diagnostic data collection for support cases

Collect comprehensive diagnostic data in one operation to reduce investigation time and ensure complete information for support teams.

The RHDH must-gather tool collects diagnostic data from Red Hat Developer Hub deployments on OpenShift Container Platform and Kubernetes platforms (AKS, EKS, GKE) with collectors specific to RHDH components.

On OpenShift Container Platform, use oc adm must-gather. On Kubernetes platforms, deploy as a Helm chart. Both methods collect the same data.

Use the RHDH must-gather tool to:

  • Prepare support cases with Red Hat Global Support Services
  • Troubleshoot production incidents
  • Conduct proactive health checks and configuration validation
  • Debug deployment issues in development or staging environments

Obtain the container image from registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9. Use the latest image version for the most up-to-date collection capabilities.

Important

These features are for Technology Preview only. Technology Preview features are not supported with Red Hat production service level agreements (SLAs), might not be functionally complete, and Red Hat does not recommend using them for production. These features provide early access to upcoming product features, enabling customers to test functionality and provide feedback during the development process.

For more information on Red Hat Technology Preview features, see Technology Preview Features Scope.

1.2. Diagnostic data types and collection scope

Review what diagnostic data the tool gathers by default and what additional data you can include for specific troubleshooting scenarios.

The RHDH must-gather tool organizes data collection into default-enabled collectors and opt-in collectors.

Important

The must-gather output might contain sensitive information such as configuration values, environment variables, and application logs. The tool automatically sanitizes known secret types, but you should always review the output and manually check for domain-specific sensitive information before sharing externally or uploading to support cases.

1.2.1. Default-enabled collectors

The following collectors run by default and you can exclude them using --without-<collector> flags to focus collection on relevant data for your deployment:

Platform collector
Collects cluster version and identifies your platform (OpenShift Container Platform, AKS, EKS, or GKE) to help support diagnose platform-specific issues.
Helm collector
Gathers Helm release information for Helm-based deployments. Exclude this collector for Operator-based deployments.
Operator collector
Collects Operator logs and custom resources for Operator-based deployments. Essential for troubleshooting Operator deployments. Exclude this collector for Helm-based deployments.
Orchestrator collector
Gathers workflow data for deployments using the Orchestrator plugin. Useful for troubleshooting workflow issues. Exclude this collector if you do not use Orchestrator to reduce collection time.
Route and Ingress collector
Gathers route definitions on OpenShift Container Platform and ingress configurations on Kubernetes platforms. Essential for troubleshooting external access and connectivity issues.
Namespace-inspect collector
Organizes namespace resources to help support teams resolve issues faster. Limit collection to specific namespaces using the --namespaces flag if needed. Keep enabled for optimal support experience.

1.2.2. Opt-in collectors

The following collectors do not run by default and you must explicitly enable them when needed:

Cluster-info collector

Collects cluster-wide state information beyond basic platform metadata.

Important

Enable only when support requests it. This collector can significantly increase collection time and impact cluster performance.

Heap dumps collector
Collects memory snapshots for diagnosing memory leaks and performance issues. Requires proper liveness probe configuration to prevent pod restarts. Enable only when troubleshooting memory issues, as heap dumps are large files (50-500 MB each).

Configure collection scope by:

  • Excluding collectors not relevant to your deployment method
  • Filtering by specific namespaces
  • Enabling optional collectors only when needed

This reduces collection time and focuses diagnostic data on relevant areas.

1.3. Run the RHDH must-gather image on OpenShift to collect deployment manifests and logs

Run the RHDH must-gather image on OpenShift using oc adm must-gather to collect deployment manifests and logs in one operation, reducing troubleshooting time and ensuring complete diagnostic information for support cases.

Prerequisites

  • OpenShift Container Platform 4.18 or later.
  • OpenShift CLI (oc) installed and authenticated to your cluster.
  • Cluster administrator access or appropriate RBAC permissions for resource inspection.
  • The RHDH deployment you want to collect diagnostics from is accessible.

Procedure

  1. Run the must-gather collection command: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest

    A collection pod starts, data collection begins, and results appear in a timestamped directory.

  2. Optional: Specify a specific image version: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:1.10.0

    The latest tag provides the most up-to-date collection capabilities and collects data from any RHDH instance regardless of deployment version.

  3. Optional: Filter collection to specific namespaces if your RHDH deployment spans many namespaces: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --namespaces rhdh-operator,rhdh-app

    Replace rhdh-operator,rhdh-app with the comma-separated list of namespaces containing your RHDH resources.

  4. Optional: Exclude collectors not relevant to your deployment method to reduce collection time: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --without-helm

    Use --without-operator for Helm-based deployments, --without-helm for Operator-based deployments, or --without-orchestrator if not using Orchestrator functionality.

  5. Monitor the collection progress. Status messages appear as data collection progresses through different collectors. Collection typically completes within 2-5 minutes, depending on the number of resources and optional collectors enabled.

Verification

  1. Locate the output directory. The output appears in ./must-gather.local.<timestamp> in your current working directory.

  2. Verify the expected directory structure: 

    $ ls must-gather.local.<timestamp>

    The output includes directories corresponding to the collectors that ran, such as helm/, operator/, and namespaces/.

  3. Verify the tool collected resource files and logs: 

    $ find must-gather.local.<timestamp> -name "*.log" | head -5
$ find must-gather.local.<timestamp> -name "*.yaml" | head -5

Troubleshooting

If the collection fails with permission errors
Verify you have cluster administrator access or the required RBAC permissions to read cluster resources, pod logs, and namespace information.
If the collection fails with "Unauthorized" errors during data collection

Long-running collections may fail when the authentication token expires. Keep the token active by running this command in a separate terminal while collection runs: 

$ while true; do oc whoami; sleep 60; done
If the collection times out
The default timeout is typically enough for most deployments. If you have several namespaces or resources, filter to specific namespaces by using the --namespaces flag.
If specific collectors fail
Check the must-gather output for error messages. Some collectors depend on specific resources existing. For example, the Operator collector requires Backstage custom resources, which exist only in Operator-based deployments.

1.4. Run the RHDH must-gather image on Kubernetes to collect deployment manifests and logs

Run the RHDH must-gather image on Kubernetes by deploying a Helm chart to collect deployment manifests and logs from AKS, EKS, or GKE deployments in one operation, reducing troubleshooting time and ensuring complete diagnostic information for support cases.

Prerequisites

  • Kubernetes cluster running on AKS, EKS, or GKE.
  • kubectl CLI authenticated to your cluster.
  • helm CLI 3 or later installed.
  • Cluster administrator access or appropriate RBAC permissions for resource inspection.
  • The RHDH deployment you want to collect diagnostics from is accessible.

Procedure

  1. Install the must-gather Helm chart: 

    $ helm upgrade --install my-rhdh-must-gather redhat-developer-hub-must-gather \
  --repo https://charts.openshift.io \
  --namespace rhdh-diagnostics \
  --create-namespace

    This creates a namespace for diagnostic collection and starts a Deployment to run the must-gather collection.

  2. Optional: Configure collection scope using Helm values.

    Create a values.yaml file to customize collection: 

    gather:
  namespaces:
    - rhdh-app
    - rhdh-plugins
  withOperator: false  # Exclude for Helm-based deployments
  withOrchestrator: false  # Exclude if not using Orchestrator
  logLevel: "debug"  # Optional: increase verbosity

    Install with the values file: 

    $ helm upgrade --install my-rhdh-must-gather redhat-developer-hub-must-gather \
  --repo https://charts.openshift.io \
  --version {product-chart-version} \
  --namespace rhdh-diagnostics \
  --create-namespace \
  -f values.yaml

  3. Wait for the must-gather pod to be ready: 

    $ kubectl wait --for=condition=ready pod \
  -l app.kubernetes.io/instance=my-rhdh-must-gather,app.kubernetes.io/component=gather \
  --timeout=3600s -n rhdh-diagnostics

    Collection typically completes within 2-5 minutes, depending on the number of resources and optional collectors enabled.

  4. Extract the collected diagnostic data from the pod: 

    $ kubectl exec deploy/my-rhdh-must-gather -c data-holder -n rhdh-diagnostics -- tar czf - -C /must-gather . > rhdh-must-gather-output.tar.gz

    This command streams the diagnostic data archive directly to your local file system.

  5. Clean up the must-gather resources: 

    $ helm uninstall my-rhdh-must-gather -n rhdh-diagnostics
$ kubectl delete namespace rhdh-diagnostics

Verification

  1. Extract the archive: 

    $ tar -xzf rhdh-must-gather-output.tar.gz

  2. Verify the output directory structure: 

    $ ls must-gather/

    The output includes directories corresponding to the collectors that ran, such as helm/, namespaces/, and platform-specific directories.

  3. Verify the tool collected resource files and logs: 

    $ find must-gather -name "*.log" | head -5
$ find must-gather -name "*.yaml" | head -5

Troubleshooting

If the job fails to complete

Check the job status and pod logs for error messages: 

$ kubectl describe job -n rhdh-diagnostics
$ kubectl logs -n rhdh-diagnostics <pod_name>
If permission errors occur during collection
Verify the service account used by the must-gather job has appropriate RBAC permissions. Installing the chart creates necessary role bindings, but some cluster configurations might require additional permissions.
If the job times out
The default job timeout is typically enough for most deployments. If you have several namespaces or resources, configure the Helm values to filter collection to specific namespaces.
If Helm repository access fails
Ensure network connectivity to https://charts.openshift.io.

1.5. Filter data scope to reduce collection time

Customize diagnostic collection scope by filtering namespaces, excluding deployment-specific collectors, and enabling optional data when needed to reduce collection time and focus on relevant data.

Prerequisites

  • Understanding of your RHDH deployment architecture.
  • Knowledge of which namespaces contain RHDH resources.
  • Familiarity with your deployment method (Helm or Operator).
  • Basic understanding of must-gather collection on your platform (OpenShift Container Platform or Kubernetes).

Procedure

  1. Identify your deployment method:

    1. Helm deployments: exclude Operator collector
    2. Operator deployments: exclude Helm collector

  2. Determine if you use Orchestrator:

    Exclude the Orchestrator collector if your deployment does not include the Orchestrator plugin or workflow capabilities.

  3. Filter collection to specific namespaces in multi-namespace deployments:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --namespaces namespace1,namespace2,namespace3

    On Kubernetes (using Helm values file): 

    gather:
  namespaces:
    - namespace1
    - namespace2
    - namespace3

    Replace namespace1, namespace2, namespace3 with the list of namespaces containing your RHDH resources.

  4. Exclude collectors not relevant to your deployment:

    For Helm-based deployments, exclude the Operator collector:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --without-operator

    On Kubernetes (using Helm values file): 

    gather:
  withOperator: false

    For Operator-based deployments, exclude the Helm collector:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --without-helm

    On Kubernetes (using Helm values file): 

    gather:
  withHelm: false

    To exclude the Orchestrator collector:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --without-orchestrator

    On Kubernetes (using Helm values file): 

    gather:
  withOrchestrator: false

    You can combine many exclusions:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --without-operator --without-orchestrator

    On Kubernetes (using Helm values file): 

    gather:
  withOperator: false
  withOrchestrator: false

  5. Optional: Configure timeouts for long-running operations in large deployments. The default timeouts are typically enough, but you can adjust them if needed:

    On OpenShift Container Platform (using environment variables in the must-gather pod spec): 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather

    On Kubernetes (using Helm values file): 

    gather:
  cmdTimeout: "60"  # seconds

    The cmdTimeout field controls how long collectors wait for commands to complete. The default varies by collector.

  6. Optional: Set log verbosity for detailed diagnostic output during collection:

    On Kubernetes (using Helm values file): 

    gather:
  logLevel: "debug"

    Valid values are info, debug, and trace (case-insensitive). The default is info. Use debug when troubleshooting must-gather collection issues.

  7. Combine configuration options for focused collection. Example for a Helm-based deployment in specific namespaces:

    On OpenShift Container Platform: 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --namespaces rhdh-prod,rhdh-plugins --without-operator

    On Kubernetes (using Helm values file): 

    gather:
  namespaces:
    - rhdh-prod
    - rhdh-plugins
  withOperator: false

Verification

After collection completes, verify that the output has only the expected data:

  1. Check which collector directories exist in the output: 

    $ ls must-gather.local.<timestamp>/  # On OpenShift
$ ls must-gather-output/             # On Kubernetes
  2. Verify the output does not include excluded collectors. For example, if you excluded the Operator collector, the output should have no operator/ directory.

  3. If you filtered by namespaces, verify that only the specified namespaces appear in the namespaces/ directory: 

    $ ls must-gather.local.<timestamp>/namespaces/  # On OpenShift
$ ls must-gather-output/namespaces/             # On Kubernetes

1.6. Collect heap dumps to diagnose memory issues

When troubleshooting memory leaks or out-of-memory errors, collect heap dumps alongside diagnostic data for detailed memory analysis and faster root cause identification.

Prerequisites

  • Understanding of basic must-gather collection on your platform.
  • Sufficient storage for heap dump files (heap dumps can be several hundred megabytes each).
Important

RHDH stops responding during heap dump collection while writing memory snapshots. Large memory footprints (1GB+) can take several seconds or minutes, causing containers to fail liveness probes and restart.

Before collecting heap dumps:

  • Increase liveness probe timeout to prevent pod restarts
  • Plan collection during maintenance windows or low-traffic periods

Procedure

  1. Choose a heap dump collection method:

    Inspector protocol (default)
    Requires no configuration changes. Connects to backend containers and collects memory snapshots automatically.
    SIGUSR2 signal
    Requires NODE_OPTIONS configuration in your deployment. Use if the inspector method fails or you prefer explicit configuration.

  2. Optional: Configure the sigusr2 method if needed. Skip this step if using the default inspector method.

    For Operator-based deployments, add the NODE_OPTIONS environment variable to your Backstage custom resource: 

    apiVersion: rhdh.redhat.com/v1alpha1
kind: Backstage
metadata:
  name: my-rhdh
spec:
  application:
    extraEnvs:
      - name: NODE_OPTIONS
        value: "--heapsnapshot-signal=SIGUSR2 --diagnostic-dir=/tmp"

    Apply the updated custom resource: 

    $ oc apply -f backstage-cr.yaml

    For Helm-based deployments, add the environment variable to your Helm values file: 

    upstream:
  backstage:
    extraEnvVars:
      - name: NODE_OPTIONS
        value: "--heapsnapshot-signal=SIGUSR2 --diagnostic-dir=/tmp"

    Update the Helm release: 

    $ helm upgrade my-rhdh redhat-developer-hub/backstage -f values.yaml -n <namespace>

    Wait for the backend pods to restart with the new configuration: 

    $ oc get pods -n <namespace> -w  # On OpenShift
$ kubectl get pods -n <namespace> -w  # On Kubernetes

  3. Increase the liveness probe timeout to prevent pod restarts during heap dump collection.

    For Operator-based deployments, patch the deployment: 

    $ oc patch deployment <deployment-name> -n <namespace> \
  -p '{"spec":{"template":{"spec":{"containers":[{"name":"backstage-backend","livenessProbe":{"failureThreshold":180}}]}}}}'

    For Helm-based deployments, update your Helm values and redeploy: 

    upstream:
  backstage:
    livenessProbe:
      failureThreshold: 180

    Wait for the pods to restart with the new configuration before proceeding.

  4. Enable heap dump collection:

    On OpenShift Container Platform (inspector method by default): 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --with-heap-dumps

    On OpenShift Container Platform (using sigusr2 method): 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --with-heap-dumps --heap-dump-method sigusr2

    On Kubernetes (using Helm values file): 

    gather:
  heapDump:
    enabled: true
    method: "inspector"  # or "sigusr2"
    timeout: "600"  # seconds, increase for large heaps if needed

  5. Optional: Combine heap dump collection with other configuration options:

    On OpenShift Container Platform (collect heap dumps from specific namespaces): 

    $ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest -- /usr/bin/gather --with-heap-dumps --namespaces rhdh-prod

    On Kubernetes (using Helm values file): 

    gather:
  namespaces:
    - rhdh-prod
  heapDump:
    enabled: true
    method: "inspector"
  withOperator: false

  6. Monitor collection progress.

    Heap dump collection typically takes 5-15 minutes depending on backend pod count and memory size.

Verification

  1. Locate heap dump files in the output directory: 

    $ find must-gather.local.<timestamp> -name "*.heapsnapshot"  # On OpenShift
$ find must-gather-output -name "*.heapsnapshot"             # On Kubernetes

    The output includes at least one .heapsnapshot file for each RHDH backend pod running during collection.

  2. Check the size of the heap dump files: 

    $ find must-gather.local.<timestamp> -name "*.heapsnapshot" -exec ls -lh {} \;  # On OpenShift
$ find must-gather-output -name "*.heapsnapshot" -exec ls -lh {} \;             # On Kubernetes

    Typical heap dump size ranges from 50-500 MB each, depending on the backend memory usage.

  3. Verify the total output size before uploading to a support case: 

    $ du -sh must-gather.local.<timestamp>  # On OpenShift
$ du -sh must-gather-output             # On Kubernetes

    Heap dumps can significantly increase the total output size. Ensure you have sufficient storage and network capacity to upload the archive to your support case.

Troubleshooting

If heap dumps are not collected

For the inspector method (default), check the must-gather pod logs for collection errors. For the sigusr2 method, verify you configured backend containers with NODE_OPTIONS including --heapsnapshot-signal=SIGUSR2 --diagnostic-dir=/tmp. 

$ oc logs -n openshift-must-gather-<hash> <must-gather-pod>  # On OpenShift
$ kubectl logs -n rhdh-diagnostics <must-gather-pod>         # On Kubernetes
If collection starts but times out before completing

You see progress messages but collection eventually times out before finishing. Check the must-gather logs for warnings about liveness probe settings: 

[WARN] Pod 'xxx' may restart during heap dump collection!
[WARN]   Current: failureThreshold=3 × periodSeconds=10s = 30s before restart
[WARN]   Required: at least 900s (HEAP_DUMP_TIMEOUT)

Set your liveness probe timeout longer than the collection time. Increase the timeout to allow collection to complete: 

$ oc patch deployment <deployment-name> -n <namespace> \
  -p '{"spec":{"template":{"spec":{"containers":[{"name":"backstage-backend","livenessProbe":{"failureThreshold":180}}]}}}}'

Wait for your pods to restart with the new configuration, then run heap dump collection again.

If collection times out

Large memory footprints (multiple gigabytes) can take 10-15 minutes to snapshot. Increase the heap dump timeout to allow more time for collection.

On OpenShift Container Platform, set the timeout using an environment variable: 

$ oc adm must-gather --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest \
  -- /usr/bin/env HEAP_DUMP_TIMEOUT=1800 /usr/bin/gather --with-heap-dumps

On Kubernetes, set the timeout in your Helm values: 

gather:
  heapDump:
    enabled: true
    timeout: "1800"
If the default inspector method fails
If your deployment uses --disable-sigusr1 in NODE_OPTIONS, you cannot use the inspector method. Either remove --disable-sigusr1 from NODE_OPTIONS, or use the sigusr2 method instead by adding --heap-dump-method sigusr2 to the gather command.
If heap dump files are empty or corrupted
For the sigusr2 method, verify your Node.js version supports the --heapsnapshot-signal flag (Node.js 12+ required). Check backend pod logs for errors during snapshot generation.

1.7.4. Platform-specific configuration

On OpenShift Container Platform

Configure must-gather using command-line flags passed to the gather script. You cannot easily set environment variables with oc adm must-gather, so use default values or the Kubernetes Helm approach if you need custom environment variables.

The following example shows an OpenShift command with multiple flags: 

$ oc adm must-gather \
  --image=registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:latest \
  -- /usr/bin/gather \
  --namespaces rhdh-prod,rhdh-plugins \
  --without-operator
On Kubernetes

Configure must-gather using a Helm values file. Create a YAML file with your configuration and reference it during Helm installation.

The following example shows a Kubernetes installation with a values file: 

$ helm install my-rhdh-must-gather \

1.8. Diagnostic data output structure and organization

Navigate the output directory structure organized by collector type, namespace, and resource type to quickly locate logs, configurations, and resource definitions.

1.8.1. Top-level directory structure

The must-gather output directory contains files and subdirectories organized by collection method and data type. The following table shows the top-level structure:

PathContents

sanitization-report.txt

Data sanitization summary and details.

all-routes.txt

All OpenShift Container Platform routes cluster-wide.

all-ingresses.txt

All Kubernetes ingresses cluster-wide.

must-gather.log

Must-gather container logs (if running in pod).

cluster-info/

Cluster-wide information (only present if you specified --cluster-info).

Deep namespace inspect data (collected by default).

platform/

Platform and infrastructure information.

helm/

Helm deployment data (native releases and standalone deployments). Only present if Helm-deployed RHDH instances are detected.

orchestrator/

Orchestrator-flavored deployment data (if detected). Only present if Orchestrator components are detected.

operator/

Operator deployment data (if RHDH operators found). Only present if RHDH operators are detected.

1.8.2. Namespace directory structure

Within the namespace-inspect/namespaces/ directory, each namespace has a subdirectory containing resource definitions and logs organized by Kubernetes API group. The following table shows the per-namespace directory organization:

SubdirectoryContents

namespaces/<namespace-name>/pods/

Pod definitions and logs. Each pod has a subdirectory containing its YAML definition and log files for each container.

namespaces/<namespace-name>/apps/

Application workload resources including Deployments, StatefulSets, DaemonSets, and ReplicaSets.

namespaces/<namespace-name>/core/

Core Kubernetes resources including Services, ConfigMaps, Secrets, and PersistentVolumeClaims.

namespaces/<namespace-name>/networking.k8s.io/

Network resources including Ingresses and NetworkPolicies.

namespaces/<namespace-name>/route.openshift.io/

Route resources (OpenShift Container Platform only).

namespaces/<namespace-name>/batch/

Batch workload resources including Jobs and CronJobs.

namespaces/<namespace-name>/autoscaling/

Autoscaling resources including HorizontalPodAutoscalers.

Note

The directory structure follows the Kubernetes API group organization as generated by oc adm inspect namespace. Custom resources appear in subdirectories named after their API groups (for example, rhdh.redhat.com/ for Backstage resources).

1.8.3. Log file naming conventions

Pod logs follow a consistent naming pattern to identify the pod and container that generated each log file: 

namespace-inspect/namespaces/<namespace-name>/pods/<pod-name>/<container-name>/logs/current.log

For example: 

namespace-inspect/namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/backstage-backend/logs/current.log

If a pod has multiple containers, each container has a separate subdirectory with its own logs: 

namespace-inspect/namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/backstage-backend/logs/current.log
namespace-inspect/namespaces/rhdh-prod/pods/backstage-developer-hub-7d8f9c5b-xk2m4/oauth2-proxy/logs/current.log

Previous container logs are available as previous.log in the same directory.

1.8.6. Heap dump file locations

When the tool collects heap dumps with the --with-heap-dumps flag, they appear within deployment-specific directories, not under namespace-inspect/namespaces/. The location depends on the installation method:

For Helm releases: 

helm/releases/ns=<namespace>/<release-name>/deployment/heap-dumps/pod=<pod-name>/container=<container-name>/heapdump-<timestamp>.heapsnapshot

For Helm standalone deployments: 

helm/standalone/ns=<namespace>/<workload-name>/deployment/heap-dumps/pod=<pod-name>/container=<container-name>/heapdump-<timestamp>.heapsnapshot

For Operator-managed instances: 

operator/backstage-crs/ns=<cr-namespace>/<cr-name>/deployment/heap-dumps/pod=<pod-name>/container=<container-name>/heapdump-<timestamp>.heapsnapshot

For example, a Helm release heap dump: 

helm/releases/ns=rhdh-prod/developer-hub/deployment/heap-dumps/pod=backstage-developer-hub-7d8f9c5b-xk2m4/container=backstage-backend/heapdump-20260430-143022.heapsnapshot

Heap dump files are binary files that can be analyzed using Chrome DevTools, Node.js heap analysis tools, or specialized memory profilers.

1.9.4. Disconnected and air-gapped environment support

The RHDH must-gather tool supports disconnected environments with image mirroring.

Disconnected environment requirements: * Mirror the must-gather image to the internal container registry. * Configure must-gather to use the mirrored image location. * Ensure network connectivity from the cluster to the internal registry.

To mirror the must-gather image, run the following commands: 

$ podman pull registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:1.10.0
$ podman tag registry.access.redhat.com/rhdh/rhdh-must-gather-rhel9:1.10.0 internal-registry.example.com/rhdh/must-gather:1.10.0
$ podman push internal-registry.example.com/rhdh/must-gather:1.10.0

To use the mirrored image on OpenShift Container Platform, run: 

$ oc adm must-gather --image=internal-registry.example.com/rhdh/must-gather:1.10.0
Note

If your internal registry requires authentication, add the registry credentials to the cluster-wide pull secret in the openshift-config namespace. The oc adm must-gather command creates temporary namespaces, so you must use cluster-wide credentials rather than namespace-specific secrets.

To use the mirrored image on Kubernetes, configure the following Helm values: 

image:
  registry: internal-registry.example.com
  repository: rhdh/must-gather
  tag: "1.10.0"
Note

If your internal registry requires authentication, create a docker registry secret and reference it in the Helm values: 

$ kubectl create secret docker-registry my-registry-secret \
  --docker-server=internal-registry.example.com \
  --docker-username=<username> \
  --docker-password=<password> \
  --namespace rhdh-diagnostics

Then add the secret to your Helm values file: 

imagePullSecrets:
  - name: my-registry-secret

Data transfer considerations for disconnected environments:

  • You must transfer collected diagnostic data from the disconnected environment to a system with internet access for support case upload.
  • Output directory size varies based on collection scope.
  • Heap dumps significantly increase output size (500 MB - 2 GB total).
  • Test collection scope in a non-production environment to estimate output size before production use.

1.9.5. Required RBAC permissions

The RHDH must-gather tool requires elevated permissions to read cluster resources, pod logs, and namespace information.

Minimum required permissions: * Read access to pods, deployments, services, configmaps in target namespaces. * Read access to pod logs. * Read access to custom resources (Backstage CRs for Operator deployments, Helm releases for Helm deployments). * Read access to routes (OpenShift Container Platform) or ingress resources (Kubernetes). * Ability to execute commands in pods (only when using --with-heap-dumps to send signals).

On OpenShift Container Platform, oc adm must-gather typically requires cluster-admin privileges, which include all necessary permissions.

On Kubernetes platforms, installing the must-gather Helm chart creates a service account with appropriate role bindings. If your cluster has restrictive security policies, you might need to create additional role bindings or use a service account with elevated permissions.

1.9.6. Known limitations and unsupported scenarios

The following scenarios have known limitations or lack support:

Unsupported platforms
Red Hat does not test or officially support Kubernetes distributions other than AKS, EKS, GKE, and OpenShift Container Platform. The tool might work on other Kubernetes distributions, but Red Hat does not guarantee functionality.
Multi-cluster deployments
The must-gather tool collects data from a single cluster. For RHDH deployments spanning multiple clusters, run must-gather separately on each cluster.
Dynamic plugin diagnostics
While the tool collects pod logs and configurations that include dynamic plugin state, it does not have specialized collectors for individual dynamic plugins. Plugin-specific diagnostics rely on application logs and ConfigMap data.
Ephemeral environments
The must-gather tool requires resources to remain stable during collection (2-15 minutes depending on scope). In highly dynamic environments with frequent pod restarts, the tool might collect incomplete data.

1.9.7. Platform-specific implementation notes

OpenShift Container Platform
The must-gather tool integrates with the OpenShift must-gather framework. The oc adm must-gather command creates a dedicated pod that runs the collection. OpenShift automatically cleans up the pod after collection completes. The tool writes output to the local file system in the must-gather.local.<timestamp> directory.
AKS
The must-gather tool deploys as a Kubernetes Job via Helm chart. Collection runs to completion, and you must extract the output from the pod by using kubectl cp. The job and pod resources persist after completion to allow data extraction, then you manually clean them up using helm uninstall.
EKS
Implementation is identical to AKS. Installing the Helm chart creates a Job that runs must-gather collection. Authentication to the EKS cluster uses kubectl with AWS credentials.
GKE
Implementation is identical to AKS. Installing the Helm chart creates a Job that runs must-gather collection. Authentication to the GKE cluster uses kubectl with GCP credentials.
